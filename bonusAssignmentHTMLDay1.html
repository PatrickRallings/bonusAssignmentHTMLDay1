Divisions of Tech

Front-End (client side)

  • Coding for most of the role and deals with the interface that people use
  • Includes building a website’s styles, such as buttons, etc
  • Common task: Updating a company’s webpage; creating an entirely new application; making sure websites are loading properly on mobile devices
  • Three big technologies:
    • HTML: Skeleton/structure of a page
    • CSS: Metaphorical skin/clothes of a webpage
    • Javascript: Deals with how a page moves; language can be used with both the browser and in the server, making it versatile
  • Front end designers usually work with the style and designs themselves, turning static images into actual functioning webpages
  • Front end engineers aren’t that involved in the design, but work with compiling the code (kind of the back end of the front end)
  • One important aspect of front-end design is cross-browser compatibility (not just differing types of browsers, but also differing stages of updated software)
  • Tools of the trade:
    • Content Management Systems, CMS
    • Libraries and Frameworks; there isn’t always a need to create from scratch when entities already exist, such as using a button that is already on the website
  • Front end is the digital representation of a company; It can have drastic implications for a brand and user experience; Builds a bridge between the customer and a product or service

Back-End (server side)

  • If the front-end is the color of the car and the design, then the back-end is how the engine and other aspects of the car actually work
  • Combination of server and database can be considered the back-end
  • Develop in back-end may be working with servers, back-end frameworks, data validation, and some integrating of user-facing elements
  • Technologies Back end uses:
    • Database (could be considered the brains)
    • Languages (such as ruby, python, php, node, and java)
    • Frameworks: Libraries of server-side programming languages that make it much easier to create and maintain coding, rather than starting from scratch each time
  • To get started in back-end development you should:
    • Choose and language and a framework that correlates to that language
    • Learn how to send content to the client
    • Learn how to take content from other areas and send them to the client, such as through APIs and HTTP
    • Connect to databases to retrieve and update data

Full-Stack is essentially putting both front-end and back-end together

  • Software developer who is writing code that works with both front and back-end as well as some vague middle areas
  • Must understand how both front and back-end work and communicate together
  • To start, don’t try and learn everything all at once; just get in and start learning one thing and that will begin helping you learn other things (also, code as much as you can to build knowledge and confidence)

Assignment: Review of Roles in Tech

Cybersecurity:

Cybersecurity is the most interesting tech “role” because of its almost limitless potential. Seemingly every aspect of technology, that manipulates data, could benefit from some form of cybersecurity. Networks and other data-driven technologies will continue to prosper from innovations, broadening the already massive scope of cybersecurity’s pervue.

Cybersecurity is a blanket term which can be applied across a large number of differing roles. In general, cybersecurity relates to programming techniques which aid in the identifying and/or securing of various digital threats. These threats can come in many forms, including SQL injecting, DDOSing, and APTs. Without the aid of cybersecurity, such attacks could cause enormous damages to organizations. One of the many interesting cybersecurity roles is known as ethical hacking. Ethical hacking is essentially the hacking of an organization without ill-intent to expose any flaws in their existing cybersecurity. However, cybersecurity roles are not always so extravagant, and can also extend to much smaller tasks, such as maintaining the security of employee badge information. Like back-end developers, roles in cybersecurity will require proficiency in any relevant coding languages. Cybersecurity developers will also need to understand the fundamentals behind network architecture and a variety of other systems in which they may be securing. Knowledge of any relevant infrastructure will be crucial for a developer to be able to identify any potential threats and secure them.

Scrum Master:

The role of scrum master is interesting due to its combination of tech, organization, and interpersonal skills. As someone who has not yet gained a deep knowledge of software development, but has strong interpersonal skills, the role of scrum master currently best suits me. As such, the role naturally draws my attention. Additionally, many of the tasks of a scrum master, such as helping others achieve objectives and conflict resolution, are tasks which I am both good at and enjoy doing.

The role of the scrum master is to lead teams towards their goals, through a variety of techniques, while utilizing the Agile project management techniques. From the business to the individual level, scrum masters work to facilitate development, clear impeding obstacles, and open lines of communication for team members. Scrum masters will work closely with product owners and help them plan future goals while maintaining Agile practices. They essentially act as the connection between the project managers and development teams. One of the most important tasks of a scrum master is that of being an effective coach for their team members. Without effective coaching, teams would lose efficiency and effectiveness. Scrum masters benefit from having a strong understanding of full-stack development and/or whatever projects their teams are working on as well as proficient knowledge of the agile project management techniques. Additionally, scrum masters should have good organization skills, be able to effectively communicate, have empathetic tendencies, and be great at conflict resolution.

Back-End Development:

The role of back-end development is interesting due to its role in providing effective programming solutions in the interest of speed and efficiency. Finding solutions to problems has always been a great area of interest and confidence for me. Furthermore, front-end coding could be ambiguous in its potential, due to the subjective nature of design and practicality, while back-end benchmarks would be based in efficiency and therefore definite, which is a quality that I value in objectives.

Back-end development, otherwise known as server-side development, is essentially the development and/or manipulation data “behind the scenes” of software. In the metaphor of a vehicle, the front-end would be the design of the car, such as the body shape, while the back-end would be the cars engine. Development in the back-end will usually involve working with databases, languages, and frameworks. It will lay the foundational code that then enables specific processes to take place based off manipulations in the client-facing front-end. The top back-end programming languages include Java, PHP, and .NET, each of which have correlating frameworks to utilize for efficiency. Back-end developers should be proficient in one or more of the aforementioned languages, or other back-end optimized languages, as well as be familiar with the relevant frameworks. Back-end developers will also likely need to know how to manage APIs, or application program interfaces, and be able to manage databases, among a variety of other concepts.

The Software Development Lifecycle (SDLC)

  • This is how they structure teams and place developers into roles
  • The two traditional ways of thinking about it are Agile and Waterfall

Waterfall Method

  • It is a methodology where each step cascades down from start to finish
  • This is the older, more traditional method of the SDLC
  • This method works best for simple projects/products which aren't expecting to evolve or change in the future
  • Roles tend to be more isolated in waterfall
  • Step 1: Defining Requirements
    • Define what will be required in order to accomplish all that you have set out to do
    • It does not have to be incredibly technical
    • The scrum master will be heavily involved in this part of the process (but will also have a small part in all other steps as well, making sure the progress is going well)
  • Step 2: Design
    • Essentially planning; includes what the actual design of the product should be as well as the engineering design of how each component works with one another
    • This is where you would start making some models on how data would flow in the final product
    • You would choose the programming language here and make all other basic foundational decisions for the product
  • Step 3: Code
    • The execution/implementation of the design step
    • Developers, once comfortable with the requirements and design, actually coding and creating the product
  • Step 4: Testing
    • As the name suggests, this is testing the code; which happens only after everything is done
    • Testing includes testing to make sure all requirements are met, not just making sure the product functions
      • Example: Making sure a page is the correct shade of a color as decided in the design stage
    • This is one of the few times where the waterfall method will allow for revisit a previous step
  • Step 5: Implementation
    • This is making sure that everything functions properly once the product actually goes live
    • This is where you would look for unanticipated circumstances
    • Sometimes the teams who actually worked on the product will not be the ones implementing it
    • This process could take a very long time, such as anywhere from 6 months to 6 years

Agile

  • Methodology that specifically allows for the ability to change and evolve as time goes on (be agile)
  • This could allow you to create a simple product with quick implementation because it allows you to easily go back and add/modify the product in the future
    • Example: If you make a website where the majority of traffic goes somewhere you don't anticipate, then you will shift the requirements to that part of the website next time you get to it in the stages of development
  • Agile sprint is simply a time frame that you are given to complete a series of tasks
  • There can be some jumping around within the differing stages; they don’t absolutely have to happen in order
  • Step 1: Requirements
    • Differs from waterfall by usually having a narrower focus on specific aspects of a product
  • Step 2: Plan
    • Focused on more of the timeframe for completing each requirement
  • Step 3: Design
    • Deciding and designing the engineering of exactly how each requirement will be met
  • Step 4: Develop
  • Step 5: Release
  • Step 6: Track and monitor

The Six Phases of the Software Development Lifecycle

  1. Analysis
    • Analysis of the big picture and the creation of the idea for a product that solves a business problem
    • Roles include: Product manager, project manager, business analyst, CTO
  2. Design
    • The phase where developers plan out the needed components and design of a product that would meet the needs of the vision created in the analysis phase
    • Roles include: system architects, UX/UI designers
  3. Development Phase
    • This is where developers begin actually creating the product
    • Roles include: front-end and back-end developers
  4. Testing Phase
    • As the name suggests, but testers do to actually fix the code in this step, they simply state the problems
    • This is where demonstrations can take place in low-level environments
  5. Deployment
    • Moving from one environment to another, such as testing to staging
    • Release process is the act of deploying a product into production and is facilitated by specific release management teams
    • Final deployment is often facilitated by someone in a Dev Ops role in tandem with the product's developers
  6. Maintenance
    • The development that occurs after a product has been deployed into the highest environment, production, and is in the hands of the end-users

How Computers Function

Computers

  • Any collection of hardware and software that process inputs and translates it into binary or manipulates it in some way

Servers

  • Servers are more or less just computers that have the ability to process information on a much faster basis than most computers
  • Generally much more sensitive to external elements
  • Housed in specialized facilities to make sure they are running cool, consistently, and securely

The Client-Server Model

  • The way in which client devices interact with servers in order to create a network
    • The client is any computer that sends and receivers messages via servers
  • This is more or less how the internet works

Computer Networking

  • Modem
    • Receives information from an Internet Service Provider (ISP) and translates it into a signal that allows your personal computer or network of computers to connect to the internet
  • Router
    • Receives the packets of information from the modem and sends them to whatever computer they correlate to
    • Takes packets from computers and sends them to into the internet through the modem
  • LAN (Local Area Network)
    • Local collection of computing devices on one network
  • WAN (Wide Area Networks)
    • Groups of interconnected LANs
  • Switches
    • Devices that physically connect computers to networks using cables

GUI (Graphical User Interface)

  • Pronounced (Gooey)
  • It's the visual interface that allows users to actually interact with the computer's software without having to enter code
  • Example: Clicking a button to bring up a menu

Internet of Things (IoT)

  • Simply physical objects which have any sort of technology that allows them to gather, connect or manipulate data in any way
  • Examples: Items with the name "Smart", such as smart fridges, smart watches, etc.

The Internet

  • Vint Cerf and Bob Khan invented one of the first iterations of the internet in their research project for the defense networks communication system
  • Began as a nationwide experimental packet network
  • Whole pieces of information move around networks and the internet divided up into differing pieces, called packets
  • Frequency Modulation
    • Different frequencies are used to signify binary values
  • Wired Connections
    • Cables, such as copper or optical, use flashes of energy or light, respectively, to signify binary values
  • Three major parts
    1. First and Last Mile
      • Everything end-users do to connect to the internet
      • Example: Emails, WIFI, etc.
    2. Internet Hubs
      • Where ISPs receive packets and figure out where they are supposed to be delivered
    3. Internet Backbone
      • Around 400 cables lay on the bottom of the oceans to create international connectivity on the internet

Packets

  • Chunks of information
  • It wouldn't be efficient for entire pieces of information to be sent between networks uninterrupted
  • Example: Information making up a picture will not be sent from google in one single string of binary, it will be broken up into packets and will arrive to their location and combine
  • Routers make sure that the correct packets of information are being sent to the correct places

Problem Solving

  • Each digit location in binary, starting from right to left, represents doubling numbers and the 0 or 1 means whether or no that number exists or not

Sizes of Data

  • Computers typically group 8 bits together at a time, creating bytes
  • 1 Byte = 8 Bits
  • Byte can represent up to 256 differing values (including 0)
  • Kilobyte = 1,000 bytes; 8,000 bits (Word document)

Representing Text

  • ASCII table showing what bits/bytes correlate with what letters or characters (standard)
    • Capitals have different correlating values than non-capitalized letters
    • Can correlate to either decimals or hexadecimal values

Representing Color

  • Most common way is to use RGB, which is three different values for red, blue, and green content within the color
  • Example: A yellow(ish) color would be represented as (255, 227, 71)

Representing Pictures

  • Each individual pixel has an RGB value

Representing Sound (MIDI format)

  • Often uses something called MIDI
  • Like RGB, it uses three different values to represent each note: (pitch, duration, relative volume)

Strings

  • Groups of characters that in themselves represent collections of data
  • Example: Words - Each character represents bits of data correlating to the ASCII table

Boolean

  • Data that represents binary values
  • Usually represented by true of false in coding
  • Example: "0<7" is Boolean because its value is true
  • Example: "0>7" is Boolean because its value is false

Basic Commands

  • "man <command>" will tell you more about the command
    • Press "q" by itself to get out of a man page
  • "pwd"
    • "Print" current directory, aka, show current place in operating system
    • It's always helpful to know what directory you are currently working in
      • Creating new files/deleting while being in the wrong directory could be a problem
    • System will always assume whatever terminal command you're giving is for the directory that you are currently in
  • "cd <FileName>"
    • "cd -" Going back one single directory
    • Will move current directory
    • Typing out partial name of file and hitting tab will fill out rest of directory name
  • "clear"
    • Clears terminal history
  • "mkdir"
    • Makes a new folder within current directory
  • "touch"
    • Creates files
    • Example: "touch Command_Line_File.txt"
  • "rm"
    • Removes (deletes) file
  • "rm -r"
    • Removes directory
  • "ls"
    • Lists directory
  • "cp"
    • Copy file
  • "chmd"
    • . = read, .. = write, ... = delete
  • "cat"
    • Shows you content of files
    • Allows you to copy to another file with cat > filename
      • If > is used while using an existing filename, then the data in that file will be deleted
    • Allows you to add text to the end of a file (append) with cat >>
      • End text input with control+d
  • Hold down option key to move directly to wherever you click

Monday
  • Git
    • Terminal-based software program that allows you to manage your version control (most common)
    • is incredibly vital to becoming a developer or anyone that works with code
    • GitHub itself will also be very useful in this class as well as in the future
    • Terminal on computer uses Git through the shell GitBash (in our case)

  • Centralized Version Control
    • Essentially the way that we manage different versions of code (especially when working with other people)
    • Versions are stored in repositories (See repository)

  • Repository
    • The area/folder where your code is stored
    • Creating a repository (repo) using Git is essentially just flagging a regular folder as a repo so that Git can know which file to tracked
    • There are local repositories, on each person's computer, and then the main repository
    • You "push" any differences whenever you make changes to some code
    • Local repositories are also updated by server (main) repository changes as well

  • GitHub
    • Where your git projects are stored
    • Cloud service for storing and accessing your own or other's repositories
    • Even major companies, like Adobe, will store their code and frameworks on GitHub

  • Local vs Remote Repository
    • Pushing is adding files to the remote repository after they have been committed in a developer's local repository
    • Committing is telling Git and the non-local repository to track a stated file
      • Basically telling Git which files you are going to push into the remote repository
    • Pushing up is updating the remote repository (master copy) after committing
    • Cloning will make a copy of the data within files as well as the repository
      • Any changes into a clone could also be pushed into the main repository eventually
    • Pull request is basically asking for the local repository to update according to the changes in the remote repository
      • Example: You've made changed to some code but don't have the access to push the changes directly into the remote repository, so you request that the remote repository highlight all of your changes and allows other developers to also look at and comment on the changes
      • GitHub is very valuable when it comes to pull requests
    • Merged is when you actually "merge" the code from your local repository and remote repository
      • This step isn't really needed if you are only working with your own code by yourself
      • You push code, but you merge branches
    • Forking is essentially a point where code is copied and then changed to go into a new direction
      • Can sometimes eventually be pushed/merged back into the original, but often is not


  • Open Source
    • When developer allows their code to be publicly visible
    • Can be a great way to crowdsource different opportunities
      • Freelancers or hobbyists can fork the open source code, creating a potentially improved product (and sometimes the originally developer will want to add that new code into the original product)
    • Open source does not always mean that you can just take that code and add to it and then make money off of that

  • Git Basic Commands
    • "git init"
      • This folder and any directories within it will be a get repo
    • "git clone"
      • You can use this command in tandem with a repo's url to copy that whole repo and place it into the folder you are currently in
    • "git add"
      • This is how you add files to be tracked before you actually commit them
      • You can still remove or change these files before you commit them
    • "git commit"
      • This is how you actually tell the computer you want some changes to be made to a repository
      • This will make any changes to your code accessible to others
      • Not actually merging any branches
    • "-m"
      • Allows you to add a message when used as an option in tandem with commit and other commands
    • "git push"
      • Actually integrating commits into a master copy or branch
      • Usually won't have the ability to push into a master repo until you are a higher-level employee
        • You can request to be able to push in these scenarios
        • Sometimes you will have the ability to push into branches instead
    • "git pull"
      • Pulls any changes from your local repository and the master (remote)
      • Doesn't actually make any changes to your code, just gives you the changes that have been made
    • "git checkout"
      • Changes which branch you are currently tracking and located in
    • "git fetch"
      • Asking for all the different branches, details, commits and etc.
        • Not actually the code
    • "git branch"
      • Just tells you the current branch you are on (like "pwd" but for branches
    • "git branch --list"
      • Tells you you're current branch as well as providing a list of all of the other branches
    • "git branch <new branch name>"
      • Creates a new branch
    • "git checkout -b <NewBranchName>"
      • Also creates a new branch
    • "git rebase"
      • Copying a code to a new branch and merging at the same time
      • Only really needed when you are on working on a branch that is not up to date on current master iteration/haven't pulled in a while
    • "git merge"
      • Actually merging/pushing code together
    • "git status"
      • Essentially asks where you are in the tracking process (what changes have been tracked or not tracked)
      • Good to use just to get an idea of where you are constantly
    • "git add <FileName>"
      • Adds a specific untracked file to be tracked from now on
    • "git add ."
      • Changes all untracked repo files to tracked files

  • Git Branches
    • Master branch
    • Hotfix branch
      • Used to add quick changes to problems in the master branch
    • Develop branch
      • Branch where constant general development is being added
    • Feature
      • Can branch off of the development branch for specific tickets
    • Release
      • Branch that connects development branches to master branches
      • Where code can be tested and finalized before being merged into the master branch
    • Head
      • The commit that you are currently viewing/working off of
    • You can see the different branches on GitHub by going to Insights -> Network
    • Practice branching in git with the website https://learngitbranching.js.org/?locale=en_US

  • Demonstration
    • ReadMe is just a file where you can document information about other files
    • GitIgnore is a file that automatically populates in a new GitHub repository where you can put other files that you cant want to connect with local repositories
    • You can choose your own license in GitHub
      • Hector usually uses the MIT license
    • Visual Studio Code is software you can use to actual create/edit code
    • Show hidden files on mac finder with shift+command+"."
    • You can get rid of a repo but just deleting the git file
    • You have to re-add files every single time that you make a change
    • Making a new branch, like using "mkdir", doesn't actually put you into the new branch, it just create it

  • The ability to scale a project/product to be larger or smaller depending on the level of demand

HTML

  • It is not actually a programming language; it is a markup language that tells web browsers how to structure web pages
  • Hypertext
    • The links that connect differing webpages
  • Markup
    • The method HTML uses for annotating text/images /etc. in a web browser
    • Example: <head>
  • Tags
    • The component that creates an element or sets apart a part of text
    • Uses < and >
    • Not case sensitive
  • Boilerplate
    • Sections of repetitive code which have little no variation from one another
  • Nesting elements
    • When you place HTML elements within one another
  • Block-Level Elements
    • Elements that separate the code in chunks of line
  • Inline Elements
    • Elements which are usually secluded to just one line of code
  • Empty Elements
    • Elements that do not need a closing tag
    • Example: "<img> or </br>"
  • Attributes
    • The extra information within an element that does not actually appear within the content
  • Anchor
    • Makes the text it encloses into a hyperlink
    • Created with <a>
      • Nested inside should be href (for link), title, for visible content, and possibly target (to open the link in a new tab)
  • Boolean Attributes
    • Attributes that are written without any values
    • Example: "disabled" attribute within an "<input type =***>"
  • Comments
    • You can make comments inside of code that will not be visible to a user by placing special characters inside of tags
    • Example: "<---This comment would not show up at all--->"
  • Head of an HTML document is not displayed in the actual loaded web page
    • It contains "<title>", which states the title for the browser tab and possibly links to CSS if it was used to style the page
    • The head is what the web browser uses to properly render the HTML document
  • Metadata
    • Data that is used to format the web page but is not visible
    • Example: <meta charset="utf-8"> sets the character encoding for the web page, allowing it to showcase pretty much any character in any language
  • Author and Description
    • Another type of metadata that can show author of the page and description of the page
      • Descriptions can help with search engine results without altering the visible web page
    • Example: "<meta name="author" content="Patrick Rallings"> for author or <meta name="description" contents="blank words">"
  • Adding Icons
    • Favicon is a classic type of icon
    • Steps for adding reference:
      • Save icon in same directory as the websites' index page using ".ico"
        • ".gif" and ".png" can also sometimes be used
      • Add reference code inside of HTML's <head>
      • Code: "<link rel="icon" href="favicon.ico" type="image/x-icon">"
  • CSS
    • Should be initially reference inside of the HTML document's header with <link>
      • "<linkrel="stylesheet"href="my-css-file.css">"
  • JavaScript
    • Should be initially referenced with "<script>" inside of the HTML document's header
      • "<scriptsrc="my-js-file.js"defer></script>"
        • Have to have defer so HTML is parsed before the script is run
